1. Product Overview
1.1 Product Summary
Language Scoop is a web-based and installable PWA for a French teacher who conducts online classes through Zoom and offline classes in a classroom. The application helps the teacher manage students, schedules, attendance, prepaid class packs, payments, homework, practice exercises, meeting links and monthly class reporting. Students receive a simple dashboard showing upcoming classes, Zoom links, class history, homework, practice exercises, payment and pack information.
1.2 Business Problem
The teacher may forget upcoming classes or fail to create and share a Zoom link.
The teacher needs a reliable record of how many classes each student completed during a calendar month.
Fees are collected in advance based on a per-class rate and a selected pack size.
A prepaid pack may span multiple months and must not reset at month-end.
Students need direct access to their schedule, meeting links, homework and remaining prepaid classes.
1.3 Product Goals
Provide one reliable operational dashboard for the teacher.
Reduce missed classes and missing meeting links.
Make attendance finalisation quick and accurate.
Keep class-pack deductions auditable and idempotent.
Show both teacher and student the number of completed classes in the current month.
Preserve permanent pack, attendance and payment history.
Operate securely on desktop and mobile as a PWA.
1.4 Non-Goals for Version 1
Automatic Zoom meeting creation through a paid API is not required.
Automatic collection of online payments is not required; payments may be recorded manually.
Full learning-management-system features such as live video, proctoring or automated grading are out of scope.
Multi-teacher marketplace functionality is out of scope unless separately approved.

2.1 Authorization Principles
Every protected API route must validate authentication, role and record ownership.
A student must never access another student's schedule, files, payments, packs, homework or attendance.
Teacher-only notes and internal financial adjustments must not be visible to students.
UUID unpredictability is not authorization.
Financial and pack-changing actions must identify the teacher who performed them.

4. Core Workflows
4.1 Student Onboarding
Teacher creates a student profile.
Teacher sets the student's fee per class and default pack size.
Teacher creates initial login credentials or invitation workflow.
Teacher creates a prepaid pack and records advance payment.
Once payment is confirmed, the pack becomes active.
Student logs in and sees their own schedule, monthly count and pack status.
4.2 Class Lifecycle
Teacher schedules an online or offline class.
For online classes, teacher adds a valid Zoom link before the configured deadline.
Class appears on teacher and student dashboards.
After the class, teacher marks attendance and confirms whether it is billable.
A qualifying billable class consumes one credit from the active pack exactly once.
The class appears in monthly and historical reporting.
4.3 Pack Renewal
System warns when two, one and zero classes remain.
When the last billable class is consumed, the pack becomes completed.
The student is shown as renewal required.
Teacher selects Renew Pack and confirms fee, size and amount.
Teacher records the advance payment.
New pack becomes active; completed pack remains in history.

6.1 Idempotency and Reversal
Finalising the same attendance twice must not deduct two credits.
Changing non-billable to billable creates exactly one deduction.
Changing billable to non-billable creates exactly one reversal.
Changing between two billable statuses must not create another deduction.
Pack balances must never silently become negative.
Attendance may still be recorded when no active pack exists, but the teacher must receive a clear renewal/outstanding warning.

7.3 Payment Rules
Payments are separate records and must not be overwritten.
A pack becomes active only after confirmed advance payment, unless the teacher uses a documented override.
Partial payments may be supported but must not be confused with a fully paid pack.
Payment records include amount, date, method, reference, status, notes and recorded-by user.
Monthly revenue received is based on confirmed payment dates, not on classes attended during the month.
Historical fee changes must not alter previous packs; each pack stores a fee-per-class snapshot.
7.4 Financial Definitions

9. Homework and Practice Exercises
9.1 Homework
Teacher can create, edit, assign and delete homework.
Assignments may include title, instructions, due date and approved attachments.
Student can view only assignments intended for them.
Submission and review may be enabled if currently supported.
Uploaded files must pass security controls described later.
9.2 Practice
Teacher can create exercises and assign them to one or more students.
Student can view and complete assigned practice.
Completion status should be visible without exposing another student's data.
French accented characters and Unicode must render correctly.
10. Zoom and Online-Class Links
Version 1 may store manually created Zoom links; automatic Zoom API creation is not required.
Only valid HTTPS meeting URLs should be accepted.
Online classes without links should trigger a prominent teacher alert.
A student may view only links for their own classes.
Clicking the link should open the Zoom app when supported or fall back to the browser.
Cancelled classes must not retain an active-looking join action.
11. Notifications and PWA
11.1 Notification Types
Upcoming class reminder.

11.2 PWA Requirements
Valid manifest, icons, start URL and display mode.
Service worker must not cache private data in a way that remains available after logout.

Offline behaviour must be defined; failed writes must not appear successful.
Updates should not leave stale financial or attendance data.

13. Security and Privacy Requirements
13.1 Production-Blocking Security Fixes
Priority
Requirement
P0
Reject dangerous uploads including HTML, SVG, JavaScript, executables, shell scripts and unknown MIME types.
P0
Set safe download headers including Content-Disposition: attachment and X-Content-Type-Options: nosniff.
P0
Require authentication and ownership authorization for every file download.
P0
Remove any JWT fallback such as dev-secret; production must fail if a secure secret is absent.
P0
Remove or disable demo/default credentials in production.
P0
Separate development, test and production databases.
P0
Fix mobile navigation so all teacher pages are accessible.
13.2 File Security
Allowlist only genuinely required formats such as PDF, JPEG, PNG, WEBP, TXT and selected Office Open XML formats.
Validate MIME type, extension, size and safe filename server-side.
Do not trust browser-provided MIME type alone.
Use authenticated fetch/Blob download or short-lived signed URLs if JWT remains in localStorage.
Do not expose file IDs publicly without authorization.
13.3 Authentication
Passwords must be securely hashed.
Login endpoint requires rate limiting and account-enumeration-safe responses.
JWTs require expiration and a strong production secret.
Long-term recommendation: migrate authentication tokens to Secure, HttpOnly, SameSite cookies.
Production errors must not expose stack traces, queries or secrets.
13.4 Security Headers
Content-Security-Policy appropriate to the application.
Strict-Transport-Security in production.
X-Content-Type-Options: nosniff.
Referrer-Policy and Permissions-Policy.
Clickjacking protection; do not use ALLOWALL unless embedding is a confirmed requirement.
13.5 Privacy
Collect only necessary student and payment data.
Do not expose personal data in URLs, logs or analytics.
Define archive and deletion behaviour.
Backups and restoration must be documented.
Legal review is required before claiming compliance with any privacy law.
14. Accessibility, Responsive Design and UX
14.1 Responsive Navigation
Keep no more than five primary bottom-navigation items.
Use an accessible More menu or drawer for Practice, Billing, Payments and other hidden pages.
Test at 320, 375, 390, 768, 1024, 1366 and 1920 pixels.
No horizontal overflow or inaccessible controls.
14.2 Accessibility
Keyboard navigation, visible focus and logical tab order.
Accessible labels for forms and icon-only buttons.
Semantic headings and tables.
Colour contrast and non-colour status indicators.
Modal focus trapping and Escape-to-close behaviour.
Support 200% zoom and reduced motion.
Correct language attributes for English and French content.
14.3 UX Principles
Today's schedule and missing Zoom links must be immediately visible.
Marking attendance should take minimal steps.
Financial labels must clearly distinguish pack value, amount paid, outstanding amount and monthly payments received.
Destructive actions require confirmation.
Empty states and error messages should guide the teacher clearly.
15. Reliability, Performance and Observability
Friendly errors for database, network, upload and notification failures.
Prevent duplicate submission with disabled states and idempotency keys.
Log technical errors safely without personal data or secrets.
Configure production error tracking, health checks and uptime monitoring.
Use pagination and indexes for students, classes, payments and history.
Document database backup frequency and tested restore procedure.
Maintain rollback instructions for code and database migrations.
16. Testing and Acceptance Criteria
16.1 Required Automated Tests
Area
Minimum tests
Security
HTML/SVG upload rejected; allowed PDF accepted; unauthenticated file request denied; cross-student access denied; missing JWT secret fails production configuration.
Authentication
Login, invalid credentials, expired token, role restrictions, student isolation and production demo-credential check.
Attendance
Single deduction, duplicate submission, billable/non-billable reversal, status changes and no negative pack balance.
Class packs
Amount, remaining classes, completion, renewal, historical fee snapshot and cross-month continuity.
Monthly counters
Current-month count, previous-month selection, month boundary and Asia/Kolkata timezone.
Payments
Confirmed, pending, partial, duplicate handling and correct linkage to pack.
UI
All teacher pages reachable on mobile; student sees own monthly counter and pack status.
Build
Lint, type check, production build and integration suite.
16.2 Key Acceptance Criteria
An eight-class pack remains active across calendar months until eight billable classes are consumed.
The monthly class count changes with the calendar month without altering pack history.
Teacher and student see the same correct current-month completed-class count for the student.
One class cannot consume two credits.
A reversal restores exactly one credit.
Completed packs remain visible in history and are never overwritten.
No student can access another student's data or files.
Dangerous files cannot execute in the application's origin.
Production build succeeds with no critical security or data-integrity blockers.
